# OpenCode 경량 오케스트레이션 플러그인 제안서

## 1. 제안 개요

본 제안은 **OpenCode 환경에서 고가 모델과 저가 모델을 역할별로 분리하여 사용하는 경량 오케스트레이션 구조**를 설계하고, 이를 **플러그인 + 에이전트 설정 + 커스텀 명령** 조합으로 구현하는 방안을 제시한다.

핵심 방향은 다음과 같다.

- **상위 계획과 최종 검수는 고성능 모델**이 담당한다.
- **세부 구현, 테스트, 반복 수정은 저가 모델**이 담당한다.
- OpenCode의 기본 기능을 최대한 활용하고, 플러그인은 **얇고 통제 가능한 계층**으로 유지한다.
- `oh-my-openagent`의 아이디어는 참고하되, 그 구조를 그대로 복제하지 않고 **비용 통제와 예측 가능성을 우선하는 구조**로 재설계한다.

이 제안의 목적은 단순히 멀티 에이전트를 붙이는 것이 아니라, **비용 효율, 디버깅 가능성, 워크플로 일관성**을 동시에 확보하는 데 있다.

---

## 2. 배경과 문제 인식

현재 사용 환경은 OpenCode이며, 상위 수준의 설계와 판단에는 **GPT-5.4 xhigh** 같은 고성능 모델을 사용할 수 있는 여건이 있다. 반면, 실제 개발 작업에서는 다음과 같은 문제가 발생한다.

1. 고가 모델을 모든 단계에 사용하면 비용이 빠르게 증가한다.
2. 단순 구현이나 반복 수정까지 고성능 모델이 처리하면 비용 대비 효율이 떨어진다.
3. 멀티 에이전트 하네스를 강하게 도입하면 자동화는 늘어나지만, 비용 흐름과 실패 원인을 추적하기 어려워진다.
4. OpenCode 외부 생태계 도구는 강력하지만, 사용자의 의도보다 도구의 철학이 앞서는 경우가 있다.

특히 `oh-my-openagent`는 OpenCode를 멀티모델 에이전트 팀처럼 사용할 수 있게 해주는 강한 구조를 제공하지만, 이 구조는 다음과 같은 특성을 가진다.

- 자동 라우팅과 역할 분리가 깊게 내장되어 있다.
- 여러 역할과 카테고리 기반 모델 매핑이 포함되어 있다.
- 빠른 생산성 체감에는 유리하지만, 비용 흐름이 불투명해질 수 있다.
- 사용자가 설계한 단순한 정책보다 도구의 기본 워크플로가 더 강하게 개입할 수 있다.

따라서 본 제안은 `oh-my-openagent`를 그대로 채택하는 대신, 다음 질문에 답하려는 시도이다.

> **"고성능 모델은 계획과 리뷰에만 쓰고, 실제 실행은 저가 모델에 맡기는 구조를 OpenCode 위에 가장 단순하고 통제 가능하게 올리는 방법은 무엇인가?"**

---

## 3. 핵심 아이디어

핵심 아이디어는 **"비싼 모델은 brain, 싼 모델은 hands"**라는 역할 분리 원칙이다.

즉,

- 문제 해석
- 작업 분해
- 아키텍처 결정
- 품질 기준 설정
- 최종 검수

이런 고판단 작업은 고가 모델이 담당하고,

- 함수 구현
- 테스트 작성
- 반복 리팩터링
- 로그 기반 수정
- 단순 패턴 변경

같은 반복 실행 작업은 저가 모델이 담당한다.

이 원칙을 OpenCode에 적용하면, 전체 구조는 다음과 같다.

```text
[Planner / Reviewer : 고성능 모델]
  - 요구사항 해석
  - 계획 수립
  - 티켓 생성
  - 아키텍처 검수
  - 최종 승인

        ↓

[Coder / Tester / Fixer : 저가 모델]
  - 코드 작성
  - 테스트 생성
  - 버그 수정
  - 반복 개선

        ↓

[Planner / Reviewer : 고성능 모델]
  - 결과 검증
  - 병합 여부 판단
  - 후속 작업 생성
```

이 구조의 본질은 **모델 성능의 계층화**가 아니라 **의사결정과 실행의 분리**이다.

## 4. 제안 목표

본 제안의 목표는 다음과 같다.

### 4.1 기능 목표
- OpenCode에서 역할별 에이전트를 구성한다.
- 상위 계획, 구현, 테스트, 검수 흐름을 분리한다.
- 티켓 단위 실행이 가능한 워크플로를 만든다.
- 작업 상태를 기록하고 재개 가능한 구조를 만든다.

### 4.2 비기능 목표
- 고가 모델 호출 지점을 제한하여 비용을 통제한다.
- 각 단계의 실패 원인을 추적 가능하게 만든다.
- 프롬프트와 정책을 코드 저장소 내부에서 명시적으로 관리한다.
- 플러그인은 최소 기능만 담당하게 하여 유지보수성을 확보한다.

### 4.3 운영 목표
- 사용자가 워크플로를 이해할 수 있어야 한다.
- 어떤 에이전트가 어떤 모델을 쓰는지 명확해야 한다.
- 자동화보다 예측 가능성을 우선한다.

## 5. 왜 `oh-my-openagent`를 그대로 쓰지 않는가

`oh-my-openagent`는 분명히 참고할 가치가 있다. 특히 다음 요소는 유용하다.

- intent gate 개념
- 역할 분리
- planner / executor / reviewer 구조
- 멀티모델 배치 아이디어
- 작업 흐름의 정형화

그러나 그대로 채택하지 않는 이유는 다음과 같다.

### 5.1 비용 통제 측면
사용자의 전략은 이미 명확하다.

- plan / 틀 / 리뷰는 고가 모델
- 멀티 에이전트 실행 / 코딩은 저가 모델

이 전략은 단순하고 예측 가능하다. 반면 `oh-my-openagent`는 자동 라우팅과 다양한 역할 계층을 통해 생산성을 높이지만, 그만큼 **비싼 모델이 어디서 얼마나 소모되는지 흐려질 수 있다.**

### 5.2 시스템 복잡도 측면
`oh-my-openagent`는 “바로 강한 자동화”에는 유리하지만, 내부 정책이 많다. 이는 다음 문제로 이어질 수 있다.

- 디버깅 포인트 증가
- 프롬프트 정책 중첩
- 사용자 의도보다 기본 워크플로 우선
- 예외 상황에서 동작 예측 어려움

### 5.3 유지보수 측면
사용자에게 필요한 것은 거대한 범용 프레임워크가 아니라, **자신의 작업 방식에 맞는 얇은 오케스트레이션 계층**이다. 따라서 `oh-my-openagent`는 “참고 아키텍처”로 활용하되, 실제 구현은 더 작은 범위로 제한하는 것이 적절하다.

## 6. 제안 아키텍처

본 제안의 아키텍처는 세 계층으로 구성된다.

### 6.1 계층 1: 정적 정책 계층
이 계층은 저장소 내부 문서와 설정으로 구성된다.

구성 요소:
- `AGENTS.md`
- `.opencode/prompts/*.md`
- `opencode.jsonc`
- `.opencode/commands/*.md`

역할:
- 프로젝트 규칙 정의
- 역할별 시스템 프롬프트 정의
- 모델 배치 정책 정의
- 명령 진입점 정의

즉, “어떤 역할이 무엇을 해야 하는가”는 이 계층에서 결정된다.

### 6.2 계층 2: 실행 계층
이 계층은 OpenCode 기본 기능이 담당한다.

구성 요소:
- primary agent
- subagent
- task 위임
- command 실행

역할:
- planner 호출
- coder / tester / fixer 호출
- 티켓별 작업 수행
- 병렬 또는 순차 실행

즉, “정책을 실제 작업으로 바꾸는 단계”가 이 계층이다.

### 6.3 계층 3: 얇은 플러그인 계층
이 계층은 최소 기능만 맡는다.

구성 요소:
- `.opencode/plugins/orchestrator.ts`
- `.opencode/plugins/budget-guard.ts`

역할:
- 위험한 실행 가드
- 상태 파일 기록
- 티켓 헬퍼 도구 제공
- 예산 정책 보조

즉, “흐름을 보조하는 계층”이지, 전체 시스템을 지배하는 계층이 아니다.

## 7. 권장 역할 구성

역할은 최소 5개로 구성하는 것이 적절하다.

### 7.1 Planner
역할:
- 요구사항 분석
- 작업 범위 정의
- 티켓 분해
- acceptance criteria 작성
- 의존성 순서 정리

권장 모델:
- GPT-5.4 xhigh 또는 이에 준하는 고성능 모델

특징:
- 파일 수정 금지
- bash 실행 금지
- 읽기 중심

### 7.2 Coder
역할:
- 단일 티켓 구현
- 지정된 파일 수정
- 요구사항에 맞는 코드 작성

권장 모델:
- 저가 코딩 특화 모델

특징:
- 티켓 범위 밖 수정 제한
- 필요한 최소 문맥만 제공

### 7.3 Tester
역할:
- 테스트 코드 작성
- 테스트 실행
- 실패 원인 요약

권장 모델:
- 빠르고 저렴한 모델

특징:
- 구현보다 검증에 집중
- 실행 결과를 구조화해 보고

### 7.4 Fixer
역할:
- 테스트 실패 수정
- 타입 오류 수정
- lint/build 오류 수정

권장 모델:
- 저가 모델

특징:
- 좁은 범위의 수정 반복에 최적화

### 7.5 Reviewer
역할:
- 결과 검토
- 설계 위반 확인
- acceptance criteria 충족 여부 판단
- 병합 또는 재작업 결정

권장 모델:
- GPT-5.4 xhigh 또는 중상급 리뷰 모델

특징:
- 가능하면 읽기 전용
- 최종 품질 게이트 역할

## 8. 권장 워크플로

권장 워크플로는 다음과 같다.

### 단계 1. Triage / Plan
사용자 요청을 Planner가 분석한다.

출력:
- scope
- non-scope
- constraints
- ticket list
- acceptance criteria
- risk notes

### 단계 2. Implement
Coder가 티켓 하나를 구현한다.

출력:
- 수정 파일 목록
- 구현 요약
- 미해결 이슈

### 단계 3. Verify
Tester가 테스트와 검증을 수행한다.

출력:
- 테스트 결과
- 실패 로그 요약
- 재현 방법

### 단계 4. Fix
Fixer가 실패 원인을 좁혀 수정한다.

출력:
- 원인
- 수정 내용
- 재검증 요청

### 단계 5. Review
Reviewer가 결과를 최종 점검한다.

출력:
- 승인 또는 반려
- 설계 위반 여부
- 후속 티켓 필요 여부

### 단계 6. Repeat if needed
반려되면 필요한 부분만 다시 실행한다.

이 구조에서 중요한 점은 **상위 모델이 모든 단계에 개입하지 않는 것**이다. 상위 모델은 계획 시점과 검수 시점, 즉 **분기점**에서만 등장해야 한다.

## 9. OpenCode에 붙이는 방식

구현 방식은 **플러그인 단독**이 아니라, 다음 세 요소의 조합이어야 한다.

### 9.1 에이전트 정의
`opencode.jsonc` 또는 `.opencode/agents/`에서 역할별 에이전트를 정의한다.

핵심 원칙:
- planner / reviewer는 고가 모델
- coder / tester / fixer는 저가 모델
- 각 역할의 툴 권한을 분리

### 9.2 커스텀 명령
`.opencode/commands/`에 워크플로 명령을 둔다.

예시:
- `/triage`
- `/implement`
- `/verify`
- `/fix`
- `/review`

핵심 원칙:
- 명령별 목적이 명확해야 한다.
- 명령별로 agent와 model 선택이 고정되어야 한다.
- 한 명령은 한 단계만 처리해야 한다.

### 9.3 얇은 플러그인
`.opencode/plugins/`에는 최소 기능만 둔다.

추천 기능:
- 위험한 명령 차단
- 티켓 상태 파일 업데이트
- 세션 중간 요약 기록
- 예산 초과 경고

비추천 기능:
- 과도한 자동 의도 분류
- 거대한 내부 오케스트레이터 재구현
- 모든 정책을 플러그인 내부에 하드코딩

## 10. 제안 파일 구조

```text
repo/
├─ AGENTS.md
├─ opencode.jsonc
└─ .opencode/
   ├─ prompts/
   │  ├─ planner.md
   │  ├─ coder.md
   │  ├─ tester.md
   │  ├─ fixer.md
   │  └─ reviewer.md
   ├─ commands/
   │  ├─ triage.md
   │  ├─ implement.md
   │  ├─ verify.md
   │  ├─ fix.md
   │  └─ review.md
   ├─ plugins/
   │  ├─ orchestrator.ts
   │  └─ budget-guard.ts
   └─ state/
      ├─ last-plan.md
      ├─ tickets.json
      └─ run-log.json
```

이 구조의 장점은 다음과 같다.

- 프롬프트, 명령, 상태가 분리된다.
- 정책과 실행이 파일 수준에서 명시된다.
- 플러그인을 제거해도 기본 워크플로는 유지된다.
- 특정 역할만 교체하기 쉽다.

## 11. 티켓 중심 설계

본 구조에서 가장 중요한 단위는 **티켓(ticket)** 이다.

좋은 티켓은 다음 요소를 가져야 한다.

- 목표
- 수정 대상 파일
- 제약 조건
- 완료 조건
- 테스트 조건
- 비범위 항목

예시:

```text
Task: auth middleware 추가

Goal:
- JWT 인증 미들웨어 구현
- /api/private/* 경로 보호

Files to modify:
- src/middleware/auth.ts
- src/routes/private.ts
- tests/auth.test.ts

Constraints:
- 기존 router 구조 유지
- 새 의존성 추가 금지
- strict mode 통과

Acceptance criteria:
- 유효 토큰이면 200
- 토큰 없으면 401
- 잘못된 토큰이면 403
- 모든 테스트 통과
```

이와 같은 티켓 중심 구조를 사용하면 저가 모델도 더 안정적으로 작동한다. 이유는 작업 범위와 성공 조건이 명확하기 때문이다.

## 12. 기대 효과

### 12.1 비용 절감
고가 모델은 planning / review에만 쓰고, 실행은 저가 모델이 담당하므로 총비용이 크게 낮아진다.

### 12.2 속도 향상
구현과 테스트는 병렬화가 가능하다. 특히 저가 모델은 대량 반복 작업에 적합하다.

### 12.3 품질 안정성
상위 모델이 acceptance criteria와 최종 리뷰를 담당하므로, 저가 모델의 편차를 흡수할 수 있다.

### 12.4 디버깅 용이성
단계가 분리되어 있으므로, 실패 원인이 plan 문제인지, 구현 문제인지, 테스트 문제인지 구분하기 쉽다.

### 12.5 유지보수성
플러그인이 얇고, 정책이 파일 기반으로 분리되어 있어 수정과 확장이 쉽다.

---

## 13. 예상 리스크와 대응 방안

### 13.1 상위 모델 호출이 너무 잦아질 위험
문제:
- 모든 중간 단계에서 planner/reviewer를 부르면 비용이 다시 증가한다.

대응:
- planner는 시작 시점에만 호출
- reviewer는 완료 직전 또는 실패 재분기 시점에만 호출

### 13.2 저가 모델의 품질 편차
문제:
- 구현 품질이 일정하지 않을 수 있다.

대응:
- 티켓을 작게 쪼갠다.
- acceptance criteria를 명확히 적는다.
- tester/fixer 단계를 분리한다.

### 13.3 과도한 자동화 유혹
문제:
- 플러그인에 계속 기능을 넣다 보면 결국 또 다른 거대 프레임워크가 된다.

대응:
- 플러그인은 가드, 상태, 소도구만 맡는다.
- 라우팅과 정책은 명령/설정에 남긴다.

### 13.4 문맥 과부하
문제:
- 저가 모델에 전체 저장소 문맥을 계속 주면 비용과 성능 모두 나빠진다.

대응:
- 티켓 단위 최소 문맥만 전달
- planner가 필요한 파일만 지정

## 14. 단계별 도입 계획

### 1단계: 최소 구조 도입
- planner / coder / reviewer 3역할 구성
- `/triage`, `/implement`, `/review` 세 명령만 먼저 구성
- 상태 파일 없이 수동 운영

### 2단계: 검증 자동화 추가
- tester / fixer 역할 추가
- `/verify`, `/fix` 명령 추가
- 기본 테스트/빌드 요약 자동 기록

### 3단계: 상태 관리 추가
- `.opencode/state/` 도입
- 티켓 상태 기록
- 세션 재개 흐름 정리

### 4단계: 예산 가드 추가
- budget-guard 플러그인 추가
- 고가 모델 호출 빈도 경고
- 위험한 명령 승인 흐름 강화

이 순서로 가면 초기에 구조를 가볍게 유지하면서도, 필요에 따라 점진적으로 고도화할 수 있다.

## 15. 최종 제안

최종적으로 제안하는 방향은 다음과 같다.

> **OpenCode의 기본 기능을 중심으로 역할 기반 멀티 에이전트 구조를 만들고, `oh-my-openagent`에서는 아이디어만 차용하되, 실제 구현은 더 얇고 통제 가능한 경량 오케스트레이션 플러그인으로 구성한다.**

구체적으로는,

- `planner / reviewer`는 고가 모델
- `coder / tester / fixer`는 저가 모델
- 워크플로는 커스텀 명령으로 분리
- 플러그인은 가드와 상태 관리만 담당
- 티켓 중심 설계로 실행 단위를 표준화

이 구조는 다음 균형을 만족한다.

- **생산성**: 멀티 에이전트 흐름 확보
- **비용**: 고가 모델 사용 최소화
- **통제성**: 모델과 역할의 경계 명확화
- **확장성**: 필요 시 단계적으로 기능 추가 가능

결론적으로, 본 제안은 OpenCode를 단순한 채팅형 코딩 도구가 아니라, **비용 효율적이고 검증 가능한 소형 멀티 에이전트 개발 환경**으로 바꾸기 위한 현실적인 설계안이다.
